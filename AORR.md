# AORR 상태 머신 설계서

이 문서는 `C:\Users\Ayw\Documents\Loop Engineering\toload.github.io` 저장소에서 GitHub Pages용 프로페셔널 웹사이트를 개발하기 위한 AORR 관점의 실행 규칙을 정의한다.

현재 저장소 기준 확인 사항:

- 현재 루트에는 `README.md`만 확인되었다.
- 웹사이트 코드와 정적 자산은 아직 없다.
- 따라서 이 저장소는 새 정적 사이트의 시작점이다.

Step 1의 게임 요구는 지렁이 게임이 아니라 사과게임으로 변경 가능하다. 다만 사과게임의 정확한 규칙은 아직 확정되지 않았으므로, 세부 규칙은 `[사람 확인 필요]`로 둔다.

## 1. Target

### 프로페셔널 웹사이트 개발 목표

- 개인의 정체성, 경력, 프로젝트, 연락처를 보여주는 프로페셔널 웹사이트를 만든다.
- 정적 HTML, CSS, JavaScript만으로 동작하게 한다.
- 외부 백엔드 서버 없이 GitHub Pages에서 바로 열리게 한다.
- 메인 소개, 게시글 목록, Games 탭을 포함한다.

### GitHub Pages 배포 목표

- 저장소의 정적 파일만으로 빌드 없이 배포 가능해야 한다.
- 루트 디렉토리에 최소한 다음 파일이 존재해야 한다.
  - `index.html`
  - `styles.css`
  - `script.js`
  - 필요 시 별도 게임 JavaScript 파일
- 상대 경로 리소스와 정적 자산만 사용한다.
- GitHub Pages에서 404 없이 열려야 한다.

### 입력 자료

- 이름
- 한 줄 소개
- 상세 소개
- 경력 요약
- 프로젝트 목록
- 연락처/외부 링크
- 프로필 이미지 유무
- 게시글 목록 메타데이터
- Games 탭 요구사항
- 사과게임 규칙과 조작 방식
- 색상/톤/브랜딩 선호

### 필수 페이지와 섹션

- 메인 페이지
  - 히어로/소개
  - 경력 또는 핵심 역량
  - 프로젝트 요약
  - 연락처 또는 외부 링크
- 게시글 목록 페이지
  - 게시글 카드 또는 리스트
  - 날짜, 제목, 요약
- Games 페이지
  - 사과게임 플레이 영역
  - 점수 표시
  - 게임 시작/재시작 UI
  - 조작 안내

### Games 탭 및 사과게임 요구사항

- 상단 내비게이션에 `Games` 탭을 둔다.
- 사과게임은 브라우저에서 단독 동작해야 한다.
- 키보드 조작을 지원해야 한다.
  - 기본 키는 `[사람 확인 필요]`
  - WASD 병행 여부는 `[사람 확인 필요]`
- 모바일 터치 조작을 지원해야 한다.
  - 스와이프 또는 온스크린 버튼 중 하나를 채택
  - 우선순위는 모바일에서 한 손 조작이 쉬운 방식
- 기본 게임 규칙에는 다음이 포함되어야 한다.
  - 사과 생성
  - 이동 또는 선택
  - 점수 증가
  - 충돌 또는 실패 조건
  - 재시작
- 사과게임의 구체 장르가 불명확하므로 아래는 확정 필요다.
  - 사과를 먹는 아케이드 게임인지 `[사람 확인 필요]`
  - 사과를 맞추거나 분류하는 게임인지 `[사람 확인 필요]`
  - 시간 제한이 있는지 `[사람 확인 필요]`
  - 최고 점수 저장 여부 `[사람 확인 필요]`
  - 사운드 사용 여부 `[사람 확인 필요]`

### 데스크톱 및 모바일 완료 기준

- 데스크톱
  - 1280px 폭 기준에서 탭, 소개, 게시글, 게임이 모두 자연스럽게 보인다.
  - 게임 조작과 점수가 한 화면에서 읽힌다.
- 모바일
  - 320px 폭에서도 가로 스크롤이 없다.
  - 탭 터치 영역이 충분히 크다.
  - 게임 조작이 손가락으로 가능하다.
  - 텍스트가 잘리거나 겹치지 않는다.

## 2. Act

### 한 번의 개발 루프에서 수행할 최소 작업

- 실패 원인 하나만 선택한다.
- 관련된 최소 파일만 수정한다.
- 한 상태에서 다음 상태로 진전할 수 있을 만큼만 구현한다.
- 기존 통과 기능을 깨지 않는지 확인한다.

### 수정 가능한 파일 범위

- `index.html`
- `styles.css`
- `script.js`
- 필요 시 게임 전용 JavaScript 파일
- 필요 시 정적 이미지/아이콘 파일

### 생성할 수 있는 파일

- `index.html`
- `styles.css`
- `script.js`
- `game.js` 또는 사과게임 전용 파일
- `assets/` 하위의 정적 이미지 파일

### 실행 가능한 로컬 검증 명령어

- 파일 존재 확인
  - `rg --files`
- HTML/CSS/JS 수동 검증
  - 로컬 에디터의 오류 표시
- 정적 서버 실행
  - `[사람 확인 필요]` 사용 가능 도구에 따라 `python -m http.server`
  - `[사람 확인 필요]` 사용 가능 도구에 따라 `npx serve`
- 브라우저 확인
  - 로컬 서버에서 열기
  - 콘솔 오류 확인
- GitHub Pages 호환성 사전 점검
  - 상대 경로만 사용하는지 확인
  - SPA 전용 라우팅 의존이 없는지 확인

## 3. Observe

- 파일 생성 여부
  - 필요한 루트 파일이 실제 존재하는지 확인
- HTML, CSS, JavaScript 오류
  - 문법 오류
  - 참조 누락
  - 런타임 예외
- 로컬 웹서버 응답
  - 루트 경로가 200 응답인지 확인
  - 정적 자산이 404가 아닌지 확인
- 브라우저 콘솔 오류
  - uncaught exception
  - resource load failure
  - layout shift 관련 이상 징후
- 데스크톱 및 모바일 화면
  - 1280px, 768px, 320px 확인
  - 탭/본문/게임이 겹치지 않는지 확인
- 키보드 및 터치 게임 조작
  - 입력이 즉시 반영되는지
  - 입력이 한 번만 처리되는지
  - 이동 또는 선택 전환이 유효한지
- GitHub Pages 호환성
  - 절대 경로 의존이 없는지
  - 외부 서버 호출이 없는지
  - 정적 파일만으로 동작하는지

## 4. Reason

실패 원인 분류 기준:

- `HTML_STRUCTURE`
  - 태그 구조 문제, 잘못된 중첩, 누락된 섹션, 잘못된 링크 구조
- `CSS_RESPONSIVE`
  - 모바일 레이아웃 깨짐, 가로 스크롤, 터치 영역 부족, 브레이크포인트 문제
- `JAVASCRIPT`
  - 일반 JS 문법 오류, 이벤트 처리 실패, DOM 참조 오류
- `GAME_LOGIC`
  - 사과게임의 생성, 점수, 실패, 재시작, 규칙 오류
- `GAME_CONTROL`
  - 키보드 또는 터치 입력이 기대대로 반응하지 않음
- `CONTENT`
  - 이름, 소개, 경력, 프로젝트, 게시글 정보가 불명확하거나 누락됨
- `TEST`
  - 검증 절차가 불완전하거나 재현성이 부족함
- `ENVIRONMENT`
  - 로컬 서버, 브라우저, 파일 경로, 실행 도구 문제
- `GITHUB_PERMISSION`
  - 저장소 권한, 토큰, Pages 권한 문제
- `DEPLOYMENT`
  - GitHub Pages 배포 실패, 404, 경로 오류, 빌드 산출물 문제
- `UNKNOWN`
  - 위 분류로도 설명되지 않는 경우

## 5. Repeat

- 한 번에 하나의 실패 원인만 수정한다.
- 관련된 최소 파일만 변경한다.
- 수정 후에는 동일한 Verifier를 다시 실행한다.
- 기존에 통과한 기능이 있으면 회귀 테스트를 함께 실행한다.
- 반복 중 새 문제가 생기면 우선순위를 다시 정한다.

## 6. Stop

다음 조건 중 하나라도 만족하면 멈춘다.

- 전체 테스트가 통과한 경우
- 최대 Retry에 도달한 경우
- 동일한 오류 fingerprint가 2회 반복된 경우
- 개인정보나 콘텐츠 확인이 필요한 경우
- GitHub 인증 또는 배포 권한 문제가 발생한 경우

## 7. Human-in-the-loop

다음 상황에서는 사람 확인이 필요하다.

- 이름, 소개, 경력, 프로젝트 등 개인 콘텐츠가 불명확한 경우
- 기존 콘텐츠 삭제가 필요한 경우
- 외부 분석 도구나 외부 서비스를 추가해야 하는 경우
- GitHub 저장소 설정을 변경해야 하는 경우
- 요구사항이 충돌하는 경우
- 사과게임의 정확한 장르와 조작 방식이 미확정인 경우
- 사운드, 최고 점수, 난이도 증가 같은 선택 기능이 결정되지 않은 경우

## 8. 실행 루프 정의

아래 상태는 `READY`, `ACTING`, `VERIFYING`, `RETRYING`, `PASSED`, `DEPLOY_READY`, `DEPLOYING`, `DEPLOYED`, `BLOCKED`, `HITL_REQUIRED`만 사용한다.

초기 상태는 `READY`다.

### 상태 전이 규칙

- `READY`는 입력이 충분하고 다음 작업을 시작할 수 있을 때 사용한다.
- `ACTING`은 파일을 수정하거나 새 파일을 만드는 중일 때 사용한다.
- `VERIFYING`은 로컬 검증과 브라우저 확인을 수행할 때 사용한다.
- `RETRYING`은 같은 실패 원인을 최소 범위로 수정하고 다시 확인할 때 사용한다.
- `PASSED`는 해당 단계의 기능이 통과했을 때 사용한다.
- `DEPLOY_READY`는 배포 직전의 검증이 끝났을 때 사용한다.
- `DEPLOYING`은 GitHub Pages 배포 작업을 수행할 때 사용한다.
- `DEPLOYED`는 배포가 정상 완료됐을 때 사용한다.
- `BLOCKED`는 환경, 권한, 기술 한계로 더 진행할 수 없을 때 사용한다.
- `HITL_REQUIRED`는 사람 결정을 기다려야 할 때 사용한다.

| GitHub Pages 호환성 검증 | 상대경로, 정적 자원, 라우팅 방식 | Pages 친화적으로 경로와 리소스를 점검한다 | 404 가능성, 절대 URL 의존, SPA 라우팅 의존을 본다 | 배포 가능성 확인 | 루트 정적 파일만으로 실행 가능 | `DEPLOY_READY` |
| 배포 | GitHub 권한, Pages 설정, 저장소 상태 | GitHub Pages로 배포한다 | 배포 상태, 접근성, 404, 정적 자산 로딩을 확인한다 | 공개 URL | 배포된 사이트가 정상 접근됨 | `DEPLOYED` 또는 `BLOCKED` |

## 11. 추천 최초 루프

가장 안전한 첫 번째 루프는 `저장소 및 기존 파일 확인`과 `프로페셔널 콘텐츠 영역`을 묶기 전의, 순수한 `Target/입력 자료 정리` 단계다.

이유:

- 코드 수정 없이 진행할 수 있다.
- 이름, 소개, 경력, 프로젝트가 확정되어야 이후 루프가 흔들리지 않는다.
- Games 탭과 사과게임은 규칙 합의가 먼저라서, 콘텐츠 합의가 끝나야 자연스럽게 이어진다.

사람 확인이 필요한 최소 항목:

- 이름
- 한 줄 소개
- 경력 공개 범위
- 프로젝트 공개 범위
- 사과게임의 정확한 장르와 조작 방식
- `게임 추가 기능:` 문구의 실제 요구사항 여부
